Compliance Matrix

A compliance matrix for the NETRINO standards model is available in the Microsoft Excel XML file standards_info/compliance/netrino_compliance.xml, which may be found relative to the LDRA tool suite installation directory.

The Netrino model makes use of the H standards, which must be enabled as described in H1. The H standards may be extended by the user to cover some other Netrino rules that are not covered in this document.

1.1.a
Mandatory
All programs shall be written to comply with the latest available ISO Standard for the C Programming Language, which is currently C99.
Mapped standards
293 S : Non ANSI/ISO construct used.  
1.1.b
Mandatory
Whenever a C++ compiler is used, appropriate compiler options shall be set to restrict the language to the latest standard C subset supported by the compiler.
Mapped standards
293 S : Non ANSI/ISO construct used.  
1.1.c
Checking
The use of proprietary compiler language keyword extensions, #pragmas, and inline assembly shall be kept to the minimum necessary to get the job done and be localized to a small number of device driver modules that interface directly to hardware.
Mapped standards
69 S : #pragma used.  
88 S : Procedure is not pure assembler.  
1.2.a
Checking
The length of all lines in a program shall be limited to a maximum of 80 characters.
Mapped standards
189 S : Input line exceeds limit.   Modifiers : 177 = 80
1.3.a
Mandatory
Braces shall always surround the blocks of code (a.k.a., compound statements), following if, else, switch, while, do, and for statements; single statements and empty statements following these keywords shall also always be surrounded by braces.
Mapped standards
11 S : No brackets to loop body (added by Testbed).  
12 S : No brackets to then/else (added by Testbed).   Modifiers : tbend = <Default>
428 S : No {} for switch (added by Testbed).  
1.3.b
Checking
Each left brace ({) shall appear by itself on the line below the start of the block it opens. The corresponding right brace (}) shall appear by itself in the same position the appropriate number of lines later in the file.
Mapped standards
188 S : { or } not on line by itself.  
301 S : } not aligned vertically below {.  
1.4.a
Mandatory
Do not rely on Cs operator precedence rules, as they may not be obvious to those who maintain the code. To aid clarity, use parentheses (and/or break long statements into multiple lines of code) to ensure proper execution order within a sequence of operations.
Mapped standards
96 S : Use of mixed mode arithmetic.   Modifiers : 267 = 250 (Default)
361 S : Expression needs brackets.   Modifiers : 119 = 0 (Default), 264 = 0 (Default), 414 = 0 (Default), 420 = 0 (Default), 421 = 0 (Default)
1.4.b
Mandatory
Unless it is a single identifier or constant, each operand of the logical && and || operators shall be surrounded by parentheses.
Mapped standards
49 S : Logical conjunctions need brackets.   Modifiers : 206 = 0 (Default)
1.5.a
Mandatory
Abbreviations and acronyms should generally be avoided unless their meanings are widely and consistently understood in the engineering community.
1.5.b
Mandatory
A table of additional project-specific abbreviations and acronyms shall be maintained in a version controlled document.
1.6.a
Mandatory
Each cast shall feature an associated comment describing how the code ensures proper behavior across the range of possible values on the right side.
1.7.a
Mandatory
The auto keyword shall not be used.
Mapped standards
464 S : Use of auto specifier.  
1.7.b
Optional
The register keyword shall not be used.
Mapped standards
84 S : Register variable declared.  
1.7.c
Mandatory
The goto keyword shall not be used.
Mapped standards
13 S : goto detected.  
1.7.d
Mandatory
The continue keyword shall not be used.
Mapped standards
32 S : Use of continue statement.  
1.7.e
Mandatory
The break keyword shall not be used outside of a switch statement.
Mapped standards
31 S : Use of break statement in loop.  
1.8.a
Mandatory
The static keyword shall be used to declare all functions and variables that do not need to be visible outside of the module in which they are declared.
Mapped standards
27 D : Variable should be declared static.  
61 D : Procedure should be declared static.  
461 S : Identifier with ambiguous linkage.  
553 S : Function and proto should both be static.  
1.8.b
Mandatory
The const keyword shall be used whenever appropriate.
Mapped standards
59 D : Parameter should be declared const.  
62 D : Pointer parameter should be declared const.  
168 S : Call by value parameter not const.  
200 S : Define used for numeric constant.  
298 S : Non const pointer to function.  
1.8.c
Mandatory
The volatile keyword shall be used whenever appropriate.
